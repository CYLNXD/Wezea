Checklist d'audit sécurité d'un site web (2026)

Un audit de sécurité web ne se résume pas à un scan de ports. Il couvre l'ensemble de la surface d'attaque : transport, authentification, email, infrastructure, code. Cette checklist structurée vous guide à travers les 8 domaines clés à vérifier pour chaque site que vous gérez.

Sommaire

  2. 2. En-têtes HTTP de sécurité
  3. 3. Authentification email
  4. 4. Infrastructure DNS

INFO
Usage de cette checklist
Cette checklist est conçue pour les audits de sécurité de sites web de PME, les audits pré-lancement, et les révisions annuelles de posture de sécurité. Elle ne remplace pas un pentest professionnel pour les sites à fort enjeu.

1. SSL / TLS

  • Certificat SSL valide et non expiré
  • Expiration à plus de 30 jours (alerte si < 30j)
  • TLS 1.2 minimum, TLS 1.3 supporté
  • TLS 1.0 et 1.1 désactivés (protocoles obsolètes)
  • Redirections HTTP → HTTPS opérationnelles sur tous les sous-domaines
  • HSTS (Strict-Transport-Security) activé avec max-age ≥ 1 an
  • Chaîne de certification complète et correcte
  • Pas de mixed content (ressources HTTP sur pages HTTPS)
# Tester la configuration TLS
openssl s_client -connect votredomaine.com:443 -tls1

# Vérifier la date d'expiration
echo | openssl s_client -connect votredomaine.com:443 2>/dev/null \
  | openssl x509 -noout -dates

# Tester HSTS
curl -sI https://votredomaine.com | grep -i strict

2. En-têtes HTTP de sécurité

Les en-têtes HTTP de sécurité protègent contre les attaques XSS, clickjacking, et injection de contenu. Vérifiez leur présence avec curl ou un outil dédié :

  • Strict-Transport-Security — force HTTPS
  • Content-Security-Policy — contrôle les sources de contenu autorisées
  • X-Frame-Options: DENY — protège contre le clickjacking
  • X-Content-Type-Options: nosniff — empêche le MIME sniffing
  • Referrer-Policy — contrôle les informations de référent
  • Permissions-Policy — restreint l'accès aux APIs navigateur (caméra, géoloc...)
# Vérifier tous les en-têtes de sécurité
curl -sI https://votredomaine.com | grep -iE "strict|content-security|x-frame|x-content|referrer|permissions"
WARN
CSP : à configurer avec précaution
Une Content-Security-Policy mal configurée peut bloquer des ressources légitimes et casser le site. Commencez en mode report-only (Content-Security-Policy-Report-Only) pour identifier les violations avant d'appliquer la politique.

3. Authentification email

  • Enregistrement SPF présent et syntaxiquement correct
  • SPF couvre tous les serveurs d'envoi légitimes
  • DKIM configuré pour le domaine principal et les sous-domaines d'envoi
  • DMARC présent avec politique p=quarantine ou p=reject
  • Tag RUA configuré pour recevoir les rapports agrégés
  • Absence d'enregistrement SPF avec +all (autorise tout le monde)
  • Enregistrement MX valide et pointant vers les bons serveurs

4. Infrastructure DNS

  • Au moins 2 serveurs DNS (redondance)
  • DNSSEC activé pour les domaines critiques
  • Aucun enregistrement CNAME pointant vers une ressource externe expirée (risk de subdomain takeover)
  • Transfert de zone DNS désactivé pour les IP non autorisées
  • Date d'expiration du domaine > 60 jours
  • Renouvellement automatique activé chez le registrar
  • 2FA activé sur le compte registrar

5. Ports exposés

Seuls les ports nécessaires au fonctionnement du service doivent être accessibles depuis Internet. Chaque port ouvert est une surface d'attaque potentielle.

  • Port 22 (SSH) — restreindre par IP ou désactiver l'accès public si possible
  • Ports 80/443 — standard, mais vérifier les versions de serveur (bannières)
  • Ports de base de données (3306 MySQL, 5432 PostgreSQL, 27017 MongoDB) — jamais accessibles depuis Internet
  • Port 3389 (RDP) — jamais exposé directement sur Internet
  • Panneau d'administration (cPanel, Plesk, phpMyAdmin) — accès restreint par IP
# Scan de ports basique (légal uniquement sur vos propres serveurs)
nmap -sV -p 1-1000 votredomaine.com

# Via nmap avec détection de version
nmap -sV --version-intensity 5 votredomaine.com

6. Vulnérabilités CVE connues

Les CVE (Common Vulnerabilities and Exposures) sont des vulnérabilités documentées dans les logiciels. Vérifier régulièrement si vos composants sont affectés est essentiel.

  • Version du serveur web (nginx, Apache) — vérifier le changelog CVE
  • Version PHP, Node.js, Python — à jour selon les branches supportées
  • CMS et plugins (WordPress, Drupal...) — versions à jour
  • Bibliothèques et dépendances — audit via npm audit, pip-audit, Composer
  • Bannières serveur désactivées (ne pas exposer les versions)

7. Contrôle des accès

  • Pages d'administration protégées par 2FA
  • Pas d'identifiants par défaut (admin/admin, root/root...)
  • Politique de mot de passe forte (minimum 12 caractères)
  • Sessions avec expiration automatique
  • Logs de connexion actifs et monitored
  • Principe du moindre privilège respecté
  • Comptes de test ou de développement supprimés en production

8. Monitoring et réponse aux incidents

  • Monitoring de disponibilité (uptime) avec alertes
  • Monitoring SSL — alerte avant expiration du certificat
  • Monitoring DNS — alerte sur tout changement d'enregistrement
  • Logs applicatifs et serveur centralisés
  • Sauvegardes testées et stockées hors site
  • Plan de réponse aux incidents documenté
  • Contact sécurité défini (qui appeler en cas d'incident ?)

Website security audit checklist (2026)

A website security audit is more than just a port scan. It covers the entire attack surface: transport, authentication, email, infrastructure, code. This structured checklist guides you through the 8 key domains to verify for every website you manage.

Table of Contents
  1. 1. SSL / TLS
  2. 2. Security HTTP headers
  3. 3. Email authentication
  4. 4. DNS infrastructure
  5. 5. Exposed ports
  6. 6. Known CVE vulnerabilities
  7. 7. Access control
  8. 8. Monitoring and response
INFO
How to use this checklist
This checklist is designed for SMB website security audits, pre-launch audits, and annual security posture reviews. It doesn't replace professional penetration testing for high-stakes websites.

1. SSL / TLS

  • Valid, non-expired SSL certificate
  • Expiration more than 30 days away (alert if < 30 days)
  • TLS 1.2 minimum, TLS 1.3 supported
  • TLS 1.0 and 1.1 disabled (obsolete protocols)
  • HTTP → HTTPS redirects working on all subdomains
  • HSTS (Strict-Transport-Security) enabled with max-age ≥ 1 year
  • Complete and correct certificate chain
  • No mixed content (HTTP resources on HTTPS pages)
# Test TLS configuration
openssl s_client -connect yourdomain.com:443 -tls1

# Check expiration date
echo | openssl s_client -connect yourdomain.com:443 2>/dev/null \
  | openssl x509 -noout -dates

# Test HSTS
curl -sI https://yourdomain.com | grep -i strict

2. Security HTTP headers

Security HTTP headers protect against XSS, clickjacking, and content injection attacks. Verify their presence with curl or a dedicated tool:

  • Strict-Transport-Security — forces HTTPS
  • Content-Security-Policy — controls authorized content sources
  • X-Frame-Options: DENY — protects against clickjacking
  • X-Content-Type-Options: nosniff — prevents MIME sniffing
  • Referrer-Policy — controls referrer information
  • Permissions-Policy — restricts access to browser APIs (camera, geolocation...)
# Check all security headers
curl -sI https://yourdomain.com | grep -iE "strict|content-security|x-frame|x-content|referrer|permissions"
WARN
CSP: configure with caution
A poorly configured Content-Security-Policy can block legitimate resources and break your site. Start in report-only mode (Content-Security-Policy-Report-Only) to identify violations before enforcing the policy.

3. Email authentication

  • SPF record present and syntactically correct
  • SPF covers all legitimate sending servers
  • DKIM configured for main domain and sending subdomains
  • DMARC present with p=quarantine or p=reject policy
  • RUA tag configured to receive aggregate reports
  • No SPF record with +all (allows everyone)
  • Valid MX record pointing to correct servers

4. DNS infrastructure

  • At least 2 DNS servers (redundancy)
  • DNSSEC enabled for critical domains
  • No CNAME records pointing to expired external resources (subdomain takeover risk)
  • DNS zone transfer disabled for unauthorized IPs
  • Domain expiration date > 60 days
  • Auto-renewal enabled at registrar
  • 2FA enabled on registrar account

5. Exposed ports

Only ports necessary for service operation should be accessible from the Internet. Each open port is a potential attack surface.

  • Port 22 (SSH) — restrict by IP or disable public access if possible
  • Ports 80/443 — standard, but check server versions (banners)
  • Database ports (3306 MySQL, 5432 PostgreSQL, 27017 MongoDB) — never accessible from Internet
  • Port 3389 (RDP) — never exposed directly to the Internet
  • Admin panel (cPanel, Plesk, phpMyAdmin) — restrict access by IP
# Basic port scan (legal only on your own servers)
nmap -sV -p 1-1000 yourdomain.com

# Via nmap with version detection
nmap -sV --version-intensity 5 yourdomain.com

6. Known CVE vulnerabilities

CVE (Common Vulnerabilities and Exposures) are documented software vulnerabilities. Regularly checking if your components are affected is essential.

  • Web server version (nginx, Apache) — check CVE changelog
  • PHP, Node.js, Python version — up-to-date per supported branches
  • CMS and plugins (WordPress, Drupal...) — versions current
  • Libraries and dependencies — audit via npm audit, pip-audit, Composer
  • Server banners disabled (don't expose versions)

7. Access control

  • Admin pages protected by 2FA
  • No default credentials (admin/admin, root/root...)
  • Strong password policy (minimum 12 characters)
  • Sessions with automatic expiration
  • Active, monitored login logs
  • Principle of least privilege respected
  • Test or development accounts removed from production

8. Monitoring and incident response

  • Uptime monitoring with alerts
  • SSL monitoring — alert before certificate expiration
  • DNS monitoring — alert on any record change
  • Centralized application and server logs
  • Tested backups stored off-site
  • Documented incident response plan
  • Security contact defined (who to call in case of incident?)
